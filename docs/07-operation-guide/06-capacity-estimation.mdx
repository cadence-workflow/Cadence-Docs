Cadence records every workflow step by calling `UpdateWorkflowExecution` internally. The estimator below counts how many times that happens for your workload and translates it into storage and compute requirements. Fill in your workflow shape and the outputs update in real time. All numbers are planning estimates; actual usage depends on deployment topology, traffic patterns, and configuration.

| Field type | Accepted formats | Examples |
|---|---|---|
| Time — avg runtime | `s`, `m`, `h`, `d`, `w` or plain number (hours) | `2h`, `30m`, `90 mins`, `1d`, `3600s`, `24` |
| Time — retention | `d`, `w`, `h` or plain number (days) | `7`, `30d`, `2w`, `48h` |

| Size in KB (signal, heartbeat, response) | `B`, `KB`, `MB`, `GB` or plain number (KB) | `1 KB`, `500 B`, `2 MB`, `1` |

## Database (Cassandra)

### Workflow shape

Four inputs define workflow volume and lifetime:

| Input | Description |
|---|---|
| `workflowsAtPeak` | How many workflows are open simultaneously at your busiest moment. DB capacity is harder to scale on the fly than compute, so size for peak. |
| `workflowsPerDay` | How many workflows start each day. Used to calculate total disk footprint. |
| `avgWorkflowRuntime` | How long a workflow runs on average. During this time it occupies hot storage in Cassandra. |
| `retentionDaysAfterCompletion` | How long Cadence keeps closed workflow history before deleting it. |

`workflowsAtPeak` and `workflowsPerDay` are independent. For a 2-hour workflow processing 10,000 orders/day, peak concurrency is `10,000 × (2 / 24) ≈ 833`, not 10,000.

### Workflow behavior

Each event type has a different write cost:

| Input | Description |
|---|---|
| `activitiesPerWorkflow` | Steps delegated to a worker process |
| `signalsPerWorkflow` | External messages sent into a running workflow |
| `timersPerWorkflow` | Built-in delays or scheduled checks |
| `childWorkflowsPerWorkflow` | Sub-workflows launched from this workflow |
| `searchAttributeUpsertsPerWorkflow` | Times the workflow updates its searchable metadata |
| `heartbeatsPerWorkflow` | Checkpoints from long-running activities |

### Payload sizes

Cadence serializes every workflow input, output, activity parameter, signal payload, and query response through a [DataConverter](/docs/concepts/data-converter) before writing it to workflow history. The default JSON converter works for most cases; the sizes you enter here should reflect what actually lands in history after serialization.

Two things affect what size to enter:
- Compression (gzip typically achieves 60–80% reduction on repetitive JSON): enter the compressed size.
- The [claim-check pattern](/docs/concepts/data-converter#claim-check): activity payloads in history become tiny references regardless of original size.

| Input | Description |
|---|---|
| `avgWorkflowPayloadSizeMB` | Input data passed when starting a workflow |
| `avgWorkflowResponseSizeKB` | Result data returned when a workflow completes |
| `avgActivityPayloadSizeMB` | Input data passed to each activity |
| `avgActivityResponseSizeKB` | Result data returned by each activity |
| `avgChildWorkflowPayloadSizeMB` | Input data passed to each child workflow |
| `avgSignalInputSizeKB` | Data carried in each signal |
| `avgHeartbeatPayloadSizeKB` | Data recorded with each activity heartbeat checkpoint |

## Formula reference

### Step 1: Count your workflow's database writes

Every time something happens in a workflow (an activity starts, a signal arrives, a timer fires), Cadence calls `UpdateWorkflowExecution` internally. Counting these calls is the first step in sizing your cluster.

The simplest possible workflow is the [Hello World](https://github.com/cadence-workflow/cadence-samples/blob/master/new_samples/hello_world/workflow.go) from cadence-samples: start, run one activity, finish.

```go
func HelloWorldWorkflow(ctx workflow.Context, input sampleInput) (string, error) {
    ao := workflow.ActivityOptions{
        ScheduleToStartTimeout: time.Minute,
        StartToCloseTimeout:    time.Minute,
    }
    ctx = workflow.WithActivityOptions(ctx, ao)

    var greetingMsg string
    err := workflow.ExecuteActivity(ctx, HelloWorldActivity, input).Get(ctx, &greetingMsg)
    if err != nil {
        return "", err
    }
    return greetingMsg, nil
}
```

For this workflow:
- `activitiesPerWorkflow = 1` (one `HelloWorldActivity`)
- `signalsPerWorkflow = 0`, `timersPerWorkflow = 0`, `childWorkflowsPerWorkflow = 0`
- `totalActions = 5 + 1*6 = 11`

The fixed `5` is what every workflow pays: start record, initial decision, close record.

```
totalActions =
    5                                       (workflow start + close, always)
  + activitiesPerWorkflow * 6
  + signalsPerWorkflow * 4
  + timersPerWorkflow * 5
  + childWorkflowsPerWorkflow * 6
  + searchAttributeUpsertsPerWorkflow * 4
  + heartbeatsPerWorkflow
```

With all behavior inputs at their defaults of 1: `totalActions = 31`

Each `UpdateWorkflowExecution` also writes task records not shown in the table, which is why the event record counts do not add up to the per-event cost.

| Event type | Cost | Key event records written |
|---|---|---|
| Workflow start/close | 5 | StartRecord, Decision, CloseRecord |
| Activity (schedule + start + complete) | 6 | Scheduled, Started, Completed, Decision |
| Signal | 4 | SignalReceiveRecord, Decision |
| Timer (fire + cancel) | 5 | Started, Fired, Decision |
| Child workflow | 6 | Initiated, Started, Completed, Decision |
| Search attribute upsert | 4 | Upserted, Decision |
| Heartbeat | 1 | RecordHeartbeat |

### Step 2: Calculate storage

Storage has two components.

**Payload data size** covers application data: workflow inputs and outputs, activity parameters, signal bodies. Cadence metadata is not counted here.

```
payloadDataSizeMB =
    avgWorkflowPayloadSizeMB
  + avgWorkflowResponseSizeKB / 1024
  + activitiesPerWorkflow * (avgActivityPayloadSizeMB + avgActivityResponseSizeKB / 1024)
  + childWorkflowsPerWorkflow * avgChildWorkflowPayloadSizeMB
  + signalsPerWorkflow * avgSignalInputSizeKB / 1024
  + heartbeatsPerWorkflow * avgHeartbeatPayloadSizeKB / 1024
```

The `/1024` terms convert KB inputs to MB. With defaults (500 KB payloads): `payloadDataSizeMB = 1.47 MB`

**Unit data size** is the Cassandra footprint for one complete workflow instance: application payload plus Cadence's record overhead across both tables.

```
unitDataSizeMB =
    historyNodeAvgRowSizeMB * (activitiesPerWorkflow + 1)
  + totalActions / 1024
  + executionsAvgRowSizeKB * 2 / 1024
```

With defaults:
- `0.6 * (1 + 1) = 1.200 MB` for history nodes. The `(activities + 1)` term models activity payload nodes (dominant by size) plus one node for workflow start and close events. Signals, timers, and decisions also write history nodes, but their entries are small and are captured by the `totalActions / 1024` term below.
- `31 / 1024 = 0.030 MB` for action records at 1 KB each.
- `7 * 2 / 1024 = 0.014 MB` for two executions rows (one for the execution record, one for timer and transfer tasks).

With defaults: `unitDataSizeMB = 1.24 MB`

**Total retained data size** is the Cassandra footprint across all live and recently closed workflows. A workflow occupies storage from the moment it starts until its retention window expires.

```
totalRetainedDataSize =
  unitDataSizeMB * workflowsPerDay * (avgWorkflowRuntime/24 + retentionDaysAfterCompletion)
```

The `(avgWorkflowRuntime/24 + retentionDays)` term is the total number of days each workflow sits in Cassandra. At Uber, production domains typically use 7–30 days depending on the workflow type.

With defaults: `totalRetainedDataSize = 1.24 * 1 * (1 + 7) = 9.92 MB`

### Step 3: Compute peak throughput

Peak throughput tells you how many `UpdateWorkflowExecution` calls per second your cluster must handle at peak — the number that drives compute sizing.

The formula applies Little's Law: if `workflowsAtPeak` workflows are always in flight and each takes `avgWorkflowRuntime` hours, then the call rate is:

```
actionsPerSec = workflowsAtPeak * totalActions / (avgWorkflowRuntime * 3600)
```

With defaults: `1 * 31 / (24 * 3600) = 0.00036 updates/sec` (near-zero load with a single workflow running all day).

A more realistic example: 10,000 concurrent workflows each running 30 minutes (0.5 hr), 31 actions each:

```
actionsPerSec = 10,000 * 31 / (0.5 * 3600) = 172 updates/sec
```

`workflowsAtPeak / workflowsPerDay` gives your environment's utilization ratio. A low ratio means most of your capacity sits idle; reach out on the [community Slack](https://cadenceworkflow.io/slack) for help improving it.

:::note
Longer `avgWorkflowRuntime` lowers `actionsPerSec` even with many concurrent workflows, because each workflow's actions are spread over more time. Shorter runtimes drive the rate up sharply. `workflowsAtPeak` does not affect storage; it only feeds into throughput and resource estimates.
:::

## Compute

The estimator scales resource requirements from a reference Cadence cluster running at **11,444** `UpdateWorkflowExecution` calls/sec. The numbers come from Uber prod11 Grafana data.

### Services and their roles

**History** executes every `UpdateWorkflowExecution`: reads state from Cassandra, validates the event, writes history and task records, releases the shard lock. CPU scales directly with throughput; it drives your core count more than any other service.

**Frontend** receives all requests from your workers, validates them, and routes to the right History shard. No state management means much lower CPU than History.

**Matching** dispatches scheduled tasks to workers. CPU-light relative to History.

**Workers** run your code. Resource requirements depend on what your activities do, not Cadence server throughput, so the estimator does not size them. A practical starting point: count concurrent activities at peak (`workflowsAtPeak × activitiesPerWorkflow`) and plan for 1 worker core per 10-50 concurrent activities, then scale based on observed CPU and latency. See the [worker configuration guide](/docs/go-client/workers) for polling and concurrency tuning.

**Cassandra** stores all history and execution state. Memory-heavy: it caches active workflows to avoid disk reads on every request. Production clusters require a minimum of **3 nodes** (replication factor 3) for fault tolerance; the estimator enforces this floor in its node count output.

### Reference clusters and scaling coefficients

Each coefficient is `(resource at reference) / 11444`. Multiply by your `actionsPerSec` to get your estimate.

| Service | Reference config | Reference total | Coefficient |
|---|---|---|---|
| History | 106 instances x 4 cores, 24 GiB each | 424 cores, 2,544 GiB RAM | 424 / 11444 cores per update/sec |
| Frontend | 63 instances x 2 cores, 12 GiB each | 126 cores, 756 GiB RAM | 126 / 11444 cores per update/sec |
| Matching | 12 instances x 6 cores, 32 GiB each | 72 cores, 384 GiB RAM | 72 / 11444 cores per update/sec |
| Cassandra | 144 nodes | 21.167 TiB RAM | 144 / 11444 nodes per update/sec |

All values round up to the next whole number. For small workloads, plan for at least one instance per service; there is fixed overhead the linear model does not capture.

## Retention and archival notes

- `retentionDaysAfterCompletion` controls how long closed workflow history stays in the primary persistence store. The total time a closed workflow remains in the database is `retentionDaysAfterCompletion` plus the DB grace period for deleted records (default 2 days for Cassandra).
- When archival is on, workflows that exceed the retention window are archived to a blob store (S3, GCS, etc.) for long-term storage. Archival storage cost is based on per-workflow size multiplied by the archival retention period. Archival is not calculated here.
- Actual behavior depends on the server-level `archival` config, domain settings, and blob store backend. See the [cluster configuration guide](/docs/operation-guide/setup) for details.

## Operational caveats

- This is a server-side estimation only. Worker capacity depends on your workflow code. Worker autoscaling, tasklist partitioning, and visibility index writes are not modeled here. See [tasklist configuration](/docs/concepts/task-lists) for partitioning guidance.
- The estimates assume mostly uniform payload and history sizes. Real workloads vary; use p95 or p99 sizes from production metrics when available.
- Cassandra and SQL backends have different data models and behave differently for the same workload. The estimates above are for Cassandra only.
- For multi-cluster or global domain setups, multiply the numbers by the number of regions.

:::note Cassandra cost and payload size
Cassandra stores each history node row as a separate SSTable entry. Larger payloads mean larger rows, which increases both disk usage and the memory Cassandra needs to cache active workflows. The estimator uses a fixed `historyNodeAvgRowSizeMB` of 0.6 MB (the Uber production average); observed production maximums reach 15 MB per history node row when activities carry large payloads. If you use the [claim-check pattern](/docs/concepts/data-converter#claim-check), history rows stay small regardless of payload size.
:::
